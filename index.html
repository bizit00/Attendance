OTP Verification Requests
Engineers request email verification before their first check-in from a new device. Click Generate & Send to create a 6-digit code and open your email client with the message pre-filled — then manually send the email to the engineer.
🔒
Device–Engineer Binding
Each engineer's phone generates a hardware-stable Hardware ID from GPU, screen, CPU and platform signals — the same physical device produces the same Hardware ID across Chrome, Firefox and Safari. If one device is used to check in more than one engineer on the same date — even from different browsers — it is automatically flagged as a suspected buddy-punch attempt.

🔐 Required Firestore Security Rules
If CRUD operations are failing, paste these rules in your Firebase Console → Firestore → Rules:
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Attendance records — anyone can write (check-in), admin can read all
    match /attendance/{doc} {
      allow create: if true;
      allow read, update, delete: if request.auth != null;
    }

    // Email ↔ Engineer ID bindings — read/write for binding checks + admin CRUD
    match /emailBindings/{email} {
      allow read, write: if true;  // Engineers need to read during check-in
    }
    match /empBindings/{empId} {
      allow read, write: if true;  // Engineers need to read during check-in
    }

    // Leave applications
    match /leaveApplications/{doc} {
      allow read, write: if true;
    }

    // Leave entitlements (per-engineer quotas)
    match /leaveEntitlements/{doc} {
      allow read: if true;
      allow write: if request.auth != null;
    }

    // ── 4. Engineer Onboarding ─────────────────────────
    match /engineerOnboarding/{docId} {
      allow read: if true;   // Engineers need to verify their own record on registration
      allow write: if request.auth != null;  // Admin only
    }

    // OTP verification requests (engineer ↔ admin handshake)
    match /otpRequests/{doc} {
      allow read, write: if true;
    }

    // Buddy-punch alerts — engineers write on detection, admin reads
    match /buddyPunchAlerts/{doc} {
      allow create: if true;
      allow read, update: if request.auth != null;
    }
  }
}

⚙️ Setup Guide — Connect Firebase (Free)

Step 1 — Create a Free Firebase Project
  1. Open console.firebase.google.com in your browser
  2. Sign in with your Google account
  3. Click "Add project" → name it Attendance System
  4. Disable Google Analytics → click "Create project"
  5. Wait ~30 seconds for it to be ready, then click "Continue"
Step 2 — Enable Firestore Database
  1. In the left sidebar, click "Firestore Database"
  2. Click "Create database"
  3. Choose "Start in production mode" (we'll add proper rules in Step 7)
  4. Select your nearest region (e.g. asia-southeast1 for Singapore) → click "Enable"
Step 3 — Enable Firebase Authentication (Required for Admin Login)
  1. In the Firebase Console left sidebar, click "Authentication"
  2. Click "Get started"
  3. Under "Sign-in method", enable "Email/Password"
  4. Go to the "Users" tab → click "Add user"
  5. Enter your admin email and a strong password (min 12 chars) → click "Add user"
  6. This email/password is what you use to log into the dashboard. No credentials are stored in the HTML file.
Step 4 — Get Your Config Keys
  1. Click the ⚙️ gear icon (top-left) → "Project settings"
  2. Scroll down to "Your apps" section
  3. Click the </> Web icon → type any app name → click "Register app"
  4. You'll see a firebaseConfig block — copy the whole object
Step 5 — Paste Config Into This File

Open attendance_system.html in Notepad (or any text editor). Find var FIREBASE_CONFIG = { near the bottom of the file and replace the placeholder values with your actual Firebase config.

var FIREBASE_CONFIG = {
  apiKey: "AIzaSy...",
  authDomain: "your-project.firebaseapp.com",
  projectId: "your-project-id",
  storageBucket: "your-project.appspot.com",
  messagingSenderId: "123456789",
  appId: "1:123456789:web:abc123"
};

⚠️ Never share your HTML file publicly with the real API key visible — restrict it in Google Cloud Console to your domain only (see security guide).

Step 6 — Update Your Work Sites

In the same file, find const WORK_SITES = [...] and edit each entry with your actual site names and GPS coordinates.

To get GPS coordinates: open Google Maps → right-click on your site location → the first line shown is latitude, longitude.

Step 7 — Set Firestore Security Rules — ALL Collections (🔒 Critical)
⚠️ If admin Edit / Delete / Reset-Devices buttons fail with "permission denied" or "Firebase write failed", it means the rules below are missing or incomplete. The app uses 5 collections — attendance, emailBindings, empBindings, leaveApplications, manpowerConfig — and all of them need rules. Paste the entire block below, don't skip any.

In Firebase Console → Firestore Database → Rules tab, replace all existing rules with:

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // ── 1. Attendance records ─────────────────────────
    match /attendance/{docId} {
      // Engineers can check in (create record) without login
      allow create: if request.resource.data.keys().hasAll([
                      'empId','empName','site','checkinTimestamp','status'])
                    && request.resource.data.empId is string
                    && request.resource.data.empId.size() > 0
                    && request.resource.data.empId.size() <= 20
                    && request.resource.data.empName is string
                    && request.resource.data.empName.size() > 0
                    && request.resource.data.empName.size() <= 60
                    && request.resource.data.status == 'in';

      // Engineers can check out (update only allowed fields)
      allow update: if request.resource.data.diff(resource.data)
                      .affectedKeys().hasOnly([
                        'checkoutTime','checkoutISO','checkoutTimestamp',
                        'checkoutLat','checkoutLng','checkoutDist',
                        'checkoutAcc','status','duration','isOvertime'])
                    && request.resource.data.status == 'out';

      // Only signed-in admin can read or delete records
      allow read, delete: if request.auth != null;
    }

    // ── 2. Email ↔ Engineer ID bindings ──────────────
    // REQUIRED for engineer registration AND admin CRUD (Edit / Delete / Reset Devices).
    // Without this block, the Engineer Accounts tab will fail with "permission denied".
    match /emailBindings/{email} {
      // Engineers (no auth) need read+create+update to register and detect conflicts
      allow read, create, update: if true;
      // Only signed-in admin can permanently delete a binding
      allow delete: if request.auth != null;
    }
    match /empBindings/{empId} {
      allow read, create, update: if true;
      allow delete: if request.auth != null;
    }

    // ── 3. Leave applications ────────────────────────
    match /leaveApplications/{docId} {
      // Any engineer can submit a leave application
      allow create: if request.resource.data.keys().hasAll([
                      'empId','empName','empEmail','leaveType',
                      'fromDate','toDate','numDays','status'])
                    && request.resource.data.empId is string
                    && request.resource.data.empId.size() > 0
                    && request.resource.data.status == 'pending';
      // Engineers can read leave records (to check status)
      allow read: if true;
      // Only signed-in admin can update (approve/reject) or delete
      allow update, delete: if request.auth != null;
    }

    // ── 4. Manpower config (required headcount per site) ──
    // Used by the Attendance Analytics → Manpower Gap section.
    match /manpowerConfig/{docId} {
      // Anyone can read so the analytics tab loads even before admin login
      allow read: if true;
      // Only signed-in admin can write the targets
      allow write: if request.auth != null;
    }

    // Leave entitlements (per-engineer quotas)
    match /leaveEntitlements/{doc} {
      allow read: if true;
      allow write: if request.auth != null;
    }

    // ── 4. Engineer Onboarding ─────────────────────────
    match /engineerOnboarding/{docId} {
      allow read: if true;   // Engineers need to verify their own record on registration
      allow write: if request.auth != null;  // Admin only
    }

    // OTP verification requests (engineer ↔ admin handshake)
    match /otpRequests/{doc} {
      allow read, write: if true;
    }

    // Buddy-punch alerts — engineers write on detection, admin reads
    match /buddyPunchAlerts/{doc} {
      allow create: if true;
      allow read, update: if request.auth != null;
    }
  }
}

✅ Paste this entire block. After clicking Publish in Firebase, wait ~30 seconds for the rules to propagate, then reload the dashboard.

🔍 How to verify rules are active:
1. Open the dashboard → Engineer Accounts tab.
2. If you see registered engineers listed, reads are working.
3. Click ✏️ Edit on any engineer, change the name, click Save.
4. If a green "✅ Account updated" toast appears, writes are working.
5. If you see "❌ Firebase write failed: permission-denied", re-check Step 7 — you missed pasting the emailBindings / empBindings blocks.

  1. Go to console.cloud.google.com

Step 10 — Verify All Collections (🩺 Troubleshooting)

Since Step 7 now contains the complete rules block for all 5 collections (attendance, emailBindings, empBindings, leaveApplications, manpowerConfig), there's nothing extra to paste here. Use this checklist if anything misbehaves:

✓ Symptom → Likely cause
"Engineer Accounts" tab is empty → emailBindings read rule missing in Step 7. Re-paste the full block.
Edit / Delete / Reset Devices fails with "permission-denied" → emailBindings or empBindings write/delete rules missing. Re-paste Step 7.
Engineer can't check in (binding write fails silently) → emailBindings create/update rules missing. Re-paste Step 7.
Leave application submit fails → leaveApplications create rule missing. Re-paste Step 7.
Approve/Reject leave fails → leaveApplications update rule needs request.auth != null. Confirm you're signed in as admin.
Rules look right but still failing → After clicking Publish, wait 30–60 seconds and hard-refresh the dashboard (Ctrl+Shift+R). Rule propagation is not instant.
Step 11 — Configure Email Notifications

To enable automatic email notifications when leaves are approved/rejected:

  1. Sign up for free at emailjs.com
  2. Go to Email Services → Add New Service → connect your Gmail account
  3. Go to Email Templates → Create New Template
  4. In your template, use exactly these variable names (copy-paste them):
    To Email field: {{to_email}}
    Subject field : {{subject}}
    Body (use any/all of these):
      {{to_name}} — Engineer name
      {{status}} — APPROVED ✅ or REJECTED ❌
      {{leave_type}} — e.g. Annual Leave
      {{from_date}} / {{to_date}}
      {{num_days}} — number of days
      {{decided_by}} — approver name
      {{reject_reason}} — rejection reason
      {{message}} — full pre-formatted message (use this alone for simplest setup)
  5. Go to Account → General to copy your Public Key
  6. Your Service ID is under Email Services (starts with service_)
  7. Your Template ID is under Email Templates (starts with template_)
  8. Go to Account → Security → add your website domain to "Allowed Origins" (e.g. https://yourname.github.io or http://localhost for local testing). This is the #1 reason emails appear sent but nothing arrives — EmailJS silently rejects requests from unregistered domains.
